# Codex・開発環境の継続改善設計

## 目的

余っているCodex利用枠を単に消費するのではなく、以後のCodex作業で発生する手戻りと、Mac開発環境の管理元調査を減らす。改善対象はCodexの恒久指示、反復運用のスキル化、ツールチェーンの可視化、Macセットアップの再現性である。

## 現状から確認できた問題

- Codexが依頼にない環境変数注入や回避策を補完し、ユーザーの意図から外れることがある。
- ユーザーが前提を否定した後も、Codexが元の前提を部分的に維持して説明を重ね、往復が増えることがある。
- Homebrew、aqua、mise、uv、rustup、アプリ同梱ランタイムが共存しているが、各ツールの正式な管理元が一覧化されていない。
- `dotfiles` はchezmoiで管理されている一方、`~/.codex/AGENTS.md` と個人Codexスキルは正本がリポジトリにない。
- `mac_setting` のセットアップ対象と、実機に導入されているツール全体との対応を機械的に検査する手段がない。

## 設計方針

### 1. Codexの恒久指示

`dotfiles` で `~/.codex/AGENTS.md` を管理する。既存ルールを維持した上で、次を明文化する。

- 明示された対象・場所・実行方法を文字どおり優先し、依頼されていない値や設定を別スコープから補完しない。
- 必須情報が欠けている場合は、勝手に秘密情報や別環境の設定を移送せず、観測した不足をそのまま報告する。
- ユーザーが前提を否定したら、その前提から導いた提案も破棄し、事実から組み立て直す。
- 複数の安全な改善を処理できる場合、ユーザーに一件を選ばせず、依存順に全件進める。
- 診断依頼では原因を先に示し、変更は依頼に含まれる場合だけ実施する。

### 2. 個人スキル

`dotfiles` で次の個人スキルを管理し、chezmoiで `~/.codex/skills` に反映する。

- `remote-ops`: ローカルとリモートの対象を明示し、複数行スクリプトはローカル一時ファイルからscpして実行・削除する。環境変数や秘密値を暗黙に移送しない。
- `codex-retrospective`: 最近のCodexタスクから誤解、手戻り、反復、環境起因の問題を抽出し、prompt、`AGENTS.md`、skill、hook、automation、環境設定のうち最小の適切な改善面へ振り分ける。

スキルは説明だけで終わらせず、入力、手順、安全境界、出力形式、検証方法を持つ。

### 3. 開発環境doctor

`mac_setting` に読み取り専用のdoctorを追加する。doctorは次を検査する。

- OS、CPU、ログインシェル
- Homebrew、aqua、mise、uv、rustup、Docker、Codexなど主要ツールの実体パスとバージョン
- 同名コマンドが複数の管理元に存在するPATH競合
- chezmoiの未反映差分
- `dotfiles` と `mac_setting` のGit状態
- Codexの恒久指示、個人スキル、Automationの存在
- セットアップで期待するツールの不足

doctorは秘密値を出力せず、変更も行わない。必須項目の欠落・競合は非ゼロ終了、参考情報は警告として扱う。人間向け出力に加え、将来のAutomationで扱える安定した行形式を用いる。

### 4. 環境の再現性

`mac_setting` にツール管理元の一覧を置き、セットアップスクリプトとdoctorが同じ一覧を参照する。Homebrewの全依存パッケージを固定するのではなく、ユーザーが明示的に利用するformula/caskだけを正本にする。

既存スクリプトは次を満たすよう整理する。

- strict modeと引用を統一する。
- 一時ファイルは予測可能な共有パスを使わず、安全な一時ディレクトリを使う。
- インストール済み判定を明示的なコマンドで行う。
- セットアップ後にdoctorを実行し、構成の欠落を検出する。
- インストールとアップグレードを混同しない。

### 5. 継続的な振り返り

週次Automationは、doctorと最近のCodexタスクを読み取り専用で確認し、改善候補を重要度順に報告する。自動で設定やコードを書き換えず、過去10回分だけ詳細をAutomation memoryに残す。既存のRenovate Automationとは役割を分ける。

## データフロー

1. `mac_setting` の管理元一覧が期待状態を定義する。
2. doctorが実機、chezmoi、Git、Codex配置を読み取る。
3. doctorは機械判定可能な `OK`、`WARN`、`FAIL` を出力する。
4. `codex-retrospective` が最近のタスクとdoctor結果をまとめ、改善面を選ぶ。
5. 人間またはAutomationが差分を確認し、必要な変更を別タスクとして実施する。

## エラー処理と安全境界

- doctorは読み取り失敗を無視せず、対象名と原因を `WARN` または `FAIL` で示す。
- コマンドが未導入でもスクリプト全体を即時終了せず、残りの検査を継続して最後に集約する。
- secret、token、cookie、認証ファイルの本文は読まない・表示しない。
- PATH競合は自動修正しない。期待管理元と実際の実体を併記する。
- Automationは報告専用とし、パッケージ更新、設定変更、commit、pushを行わない。

## テストと検証

- doctorの各検査をシェル関数として分離し、一時PATHとfixtureを使って成功、欠落、競合をテストする。
- すべてのbash/zshスクリプトに構文検査を行う。
- GitHub Actionsでdoctorのfixtureテストと構文検査を実行する。外部Actionはcommit SHAで固定する。
- `chezmoi diff` で、Codex恒久指示と個人スキル以外に意図しない変更がないことを確認する。
- 実機でdoctorを実行し、既知の管理元とPATHが正しく報告されることを確認する。
- 個人スキルは代表プロンプトでトリガーと安全境界を検証する。

## 今回行わないこと

- パッケージやOSの一括アップグレード
- PATH競合の自動修正
- secretや認証情報のdotfiles管理
- 既存Automationの役割変更
- doctor結果に基づく無承認の外部操作
